基础实验：LVGL程序开发——WhackMole实验报告
一、实验目的
基础实验围绕openvela平台下的LVGL图形界面程序开发展开，通过修改LCD、触摸屏驱动并运行MyWhackMole打地鼠游戏，完成基础图形交互功能和进阶外设交互功能。通过本次实验，主要实现以下目的：
掌握LCD显示驱动的基本修改方法，理解Framebuffer、RGB565像素格式和横屏显示适配对屏幕显示效果的影响，使开发板屏幕能够正常显示LVGL界面和MyWhackMole游戏画面
掌握触摸屏输入驱动的基本修改方法，理解触摸坐标上报、轮询/中断方式和坐标变换的作用，使触摸位置能够与屏幕显示位置对应，实现正常的人机交互
掌握openvela中LVGL应用程序的配置、编译、打包、烧录和串口运行流程，能够将MyWhackMole程序正确加入工程并在开发板上运行
能够使用LVGL图形库API实现WhackMole基础游戏，包括游戏界面、地鼠随机出现与消失、触摸击中判断、分数显示、倒计时显示和游戏结束提示等功能
通过阅读MyWhackMole程序代码，分析游戏难度与地鼠出现间隔、停留时间、游戏时长、计分规则等因素之间的关系，理解如何通过修改参数或逻辑改变游戏难度
在基础游戏功能上完成进阶交互功能，包括击中地鼠时显示视觉特效、播放hit.wav音效、控制LED1闪烁，以及按下K1键开始游戏，从而加深对openvela图形界面、声卡、LED、按键和多任务协作开发流程的理解

三、实验原理
基础实验是在openvela平台上基于LVGL实现MyWhackMole打地鼠游戏。实验整体包括LCD显示、触摸输入、游戏逻辑、音效、LED和按键交互等内容
(1) LVGL是面向嵌入式设备的图形界面库，可以创建按钮、标签、图片、动画等界面对象。本实验使用LVGL创建游戏背景、地鼠洞、地鼠对象、分数标签、倒计时标签和开始按钮，并通过事件回调和定时器实现点击判断、计时计分和地鼠随机出现
(2) LCD的显示是依赖于Framebuffer的。Framebuffer是保存整帧图像像素数据的内存区域，LVGL将界面内容写入Framebuffer后，由LCD驱动发送到屏幕显示。本次实验LCD原始分辨率为320×480，像素格式为RGB565，每个像素占16 bit。由于SPI发送数据时存在高低字节顺序问题，需要在LCD驱动中交换RGB565数据的高低字节；同时为了横屏显示游戏，还需要修改LCD方向参数和分辨率配置
(3) 触摸屏驱动负责把用户触摸转换为坐标输入。本实验使用FT5X06触摸屏驱动，并将触摸设备作为LVGL的输入设备使用。为了提高触摸响应效果，需要关闭FT5X06_POLLMOD，并启用LVGL的NuttX触摸屏支持。横屏后还需要进行坐标交换和坐标修正，使触摸位置与屏幕显示位置对应
(4) MyWhackMole游戏的核心逻辑是：游戏开始后，定时器控制地鼠随机出现和倒计时变化；用户点击屏幕时，事件回调函数判断是否击中地鼠；击中后更新分数并刷新界面。游戏难度可以通过地鼠出现间隔、停留时间、游戏总时长、计分规则等参数进行调整
(5) 进阶功能主要是在击中地鼠后增加反馈效果。视觉特效通过LVGL创建半透明框或高亮对象实现；音效通过播放res文件夹中的hit.wav实现；LED1闪烁通过控制开发板LED实现；K1开始游戏则通过读取开发板按键输入并触发start_game逻辑实现。为了避免影响界面刷新，音效、LED和按键检测适合使用任务、线程或标志位方式配合主界面运行
通过本次实验，可以把LCD、触摸屏、LVGL图形界面、声卡、LED、按键和多任务协作结合起来，理解openvela上嵌入式图形交互程序的基本开发流程

(二) 修改LCD显示驱动
1. 打开LCD驱动文件spi_lcd_fb.c，文件路径如下：
/home/ubuntu/vela-opensource/nuttx/drivers/video/spi_lcd_fb.c
本实验LCD通过SPI接口发送RGB565像素数据。由于发送顺序与Framebuffer中存储的高低字节顺序相反，需要在发送前对每个像素的两个字节进行交换！
首先在spi_lcd_updatearea函数定义之前添加RGB565字节交换函数sw_rgb565_swap：

static void sw_rgb565_swap(void * buf, uint32_t buf_size_px)
{
    uint32_t u32_cnt = buf_size_px / 2;
    uint16_t * buf16 = buf;
    uint32_t * buf32 = buf;
    while(u32_cnt >= 8) {
        buf32[0] = ((buf32[0] & 0xff00ff00) >> 8) | ((buf32[0] & 0x00ff00ff) << 8);
        buf32[1] = ((buf32[1] & 0xff00ff00) >> 8) | ((buf32[1] & 0x00ff00ff) << 8);
        buf32[2] = ((buf32[2] & 0xff00ff00) >> 8) | ((buf32[2] & 0x00ff00ff) << 8);
        buf32[3] = ((buf32[3] & 0xff00ff00) >> 8) | ((buf32[3] & 0x00ff00ff) << 8);
        buf32[4] = ((buf32[4] & 0xff00ff00) >> 8) | ((buf32[4] & 0x00ff00ff) << 8);
        buf32[5] = ((buf32[5] & 0xff00ff00) >> 8) | ((buf32[5] & 0x00ff00ff) << 8);
        buf32[6] = ((buf32[6] & 0xff00ff00) >> 8) | ((buf32[6] & 0x00ff00ff) << 8);
        buf32[7] = ((buf32[7] & 0xff00ff00) >> 8) | ((buf32[7] & 0x00ff00ff) << 8);
        buf32 += 8;
        u32_cnt -= 8;
    }
    while(u32_cnt) {
        *buf32 = ((*buf32 & 0xff00ff00) >> 8) | ((*buf32 & 0x00ff00ff) << 8);
        buf32++;
        u32_cnt--;
    }
    if(buf_size_px & 0x1) {
        uint32_t e = buf_size_px - 1;
        buf16[e] = ((buf16[e] & 0xff00) >> 8) | ((buf16[e] & 0x00ff) << 8);
    }
}

然后修改spi_lcd_updatearea函数的代码，使SPI发送交换顺序的数据。代码如下：

static int spi_lcd_updatearea(struct fb_vtable_s *vtable, FAR const struct fb_area_s *area)
{
  struct spi_dev_s *spi = g_spi_lcd_fb->spi;
  int stride = g_spi_lcd_fb->planeinfo.stride;
  int width_byte = area->w * (g_spi_lcd_fb->planeinfo.bpp >> 3);
  unsigned char *fb = g_spi_lcd_fb->planeinfo.fbmem + area->y * stride + area->x * (g_spi_lcd_fb->planeinfo.bpp >> 3);
  static uint8_t *swapped_datas;
  if (!swapped_datas)
  {
    swapped_datas = malloc(stride);
  }
  LCD_SetWindows(area->x, area->y, area->x + area->w - 1, area->y + area->h - 1);
  lcd_lock();
  LCD_SetDataLine();
  SPI_SELECT(spi, 0, true);
  for (int y = area->y; y < area->y + area->h; y++)
  {
    memcpy(swapped_datas, fb, width_byte);
    sw_rgb565_swap(swapped_datas, area->w);
    SPI_SNDBLOCK(spi, swapped_datas, width_byte);
    fb += stride;
  }
  SPI_SELECT(spi, 0, false);
  lcd_unlock();
  return 0;
}

(四) LCD 与触摸屏横屏适配
完成 LCD 基础显示修复和触摸屏驱动修改后，为了让 MyWhackMole 游戏界面更适合横屏显示，还需要继续对 LCD 显示方向、Framebuffer 分辨率以及触摸屏坐标映射进行适配。该步骤的核心目的是使屏幕显示方向变为 480×320 横屏，并保证触摸位置与界面响应位置一致！
1. 修改 LCD 横屏显示方向
打开 LCD 驱动文件 spi_lcd_fb.c，文件路径如下：
/home/ubuntu/vela-opensource/nuttx/drivers/video/spi_lcd_fb.c
在文件中搜索 LCD_WritePara(0x48)，该参数用于设置 LCD 的扫描方向。为了将屏幕从竖屏显示调整为横屏显示，需要将实际生效初始化分支中的参数修改为 0xe8
原代码为：
LCD_WritePara(0x48)
修改为：
LCD_WritePara(0xe8)
注意：所有的搜索结果都要修改！！！

2. 修改 LCD 显示范围参数
继续在 spi_lcd_fb.c 中搜索 set_screen_size 附近的代码。原来 LCD 按 320×480 竖屏设置列地址和行地址，其中 0x3f 对应 320 - 1，0xdf 对应 480 - 1。横屏显示时需要将显示范围调整为 480×320，因此需要交换这两个范围参数
原代码为：
LCD_WriteCmd(0x2a);
LCD_WritePara(0x00);
LCD_WritePara(0x00);
LCD_WritePara(0x01);
LCD_WritePara(0x3f);
LCD_WriteCmd(0x2b);
LCD_WritePara(0x00);
LCD_WritePara(0x00);
LCD_WritePara(0x01);
LCD_WritePara(0xdf);
修改为：
LCD_WriteCmd(0x2a);
LCD_WritePara(0x00);
LCD_WritePara(0x00);
LCD_WritePara(0x01);
LCD_WritePara(0xdf);
LCD_WriteCmd(0x2b);
LCD_WritePara(0x00);
LCD_WritePara(0x00);
LCD_WritePara(0x01);
LCD_WritePara(0x3f);
这样设置后，LCD 的列方向范围变为 0～479，行方向范围变为 0～319，与横屏下的 480×320 分辨率对应

3. 修改 Framebuffer 分辨率
在 spi_lcd_fb_register 函数中找到 Framebuffer 分辨率设置代码。原来系统按竖屏方式设置分辨率：
fb->videoinfo.xres = 320;
fb->videoinfo.yres = 480;
为了适配 WhackMole 横屏游戏界面，需要修改为：
fb->videoinfo.xres = 480;
fb->videoinfo.yres = 320;
修改后，Framebuffer 的逻辑分辨率与 LCD 横屏显示方向一致，LVGL 在绘制界面时会按照 480×320 的横屏尺寸进行布局

4. 修改触摸屏横屏坐标映射
LCD 改为横屏后，触摸屏上报的原始坐标方向可能仍然与竖屏方向对应。如果不进行坐标适配，可能出现手指点击位置和界面响应位置不一致的问题。因此需要继续修改触摸屏驱动 ft5x06.c，使触摸坐标与横屏后的 LCD 显示方向保持一致！
打开触摸屏驱动文件：
/home/ubuntu/vela-opensource/nuttx/drivers/input/ft5x06.c
在文件中搜索 #ifdef CONFIG_FT5X06_SWAPXY 附近的代码，将触摸点的 x 坐标改为由触摸芯片的 Y 坐标转换得到，并根据横屏宽度 480 进行方向修正
原代码为：
x = TOUCH_POINT_GET_Y(touch[0]);
修改为：
x = 480 - TOUCH_POINT_GET_Y(touch[0]);
同时，将多点触摸处理中的坐标映射代码：
point[i].x = TOUCH_POINT_GET_Y(touch[i]);
修改为：
point[i].x = 480 - TOUCH_POINT_GET_Y(touch[i]);
这样处理后，启用 X/Y 坐标交换时，触摸屏上报到 LVGL 的 x 坐标会根据横屏宽度进行反向修正，使横屏显示下的触摸位置更加准确
  

(九) 阅读代码并分析游戏难度调整方法
通过阅读 MyWhackMole.c 文件可以看出，本游戏的难度并不是由单一变量决定的，而是由游戏总时长、地鼠刷新速度、每次出现的地鼠数量、地鼠显示时间、地鼠位置分布、点击区域大小以及计分方式等多个因素共同决定的。通过修改这些参数，可以比较方便地调整游戏的整体难度和节奏！
1. 修改游戏总时长
游戏总时间主要由 GAME_TIME 宏定义控制，代码中默认设置为 30 秒：
#define GAME_TIME 30
static int game_time = GAME_TIME
这里的 GAME_TIME 表示一局游戏持续的总时间。如果把 GAME_TIME 改小，例如改成 20，游戏时间会缩短，玩家需要在更短时间内获得分数，游戏节奏会更紧张，难度也会提高。如果把 GAME_TIME 改大，例如改成 60，玩家有更多时间反应和操作，整体难度就会降低
因此，GAME_TIME 主要影响的是一局游戏的节奏压力。时间越短，玩家越容易紧张；时间越长，玩家操作空间越大
2. 修改地鼠刷新速度
地鼠出现的速度主要由 mole_timer 定时器控制。在 start_game() 函数中，程序创建了地鼠刷新定时器：
mole_timer = lv_timer_create(pop_random_mole, 1000, NULL)
其中 1000 表示每 1000 ms 调用一次 pop_random_mole() 函数，也就是每隔 1 秒刷新一次地鼠。如果把这个值改成 700，地鼠刷新速度会变快，玩家需要更快观察和点击，难度会提高。如果改成 1500，地鼠停留时间更长，玩家有更多反应时间，难度会降低
另外，在 pop_random_mole() 函数中，程序还根据剩余时间动态调整刷新周期：
if (game_time < 40) {
    lv_timer_set_period(timer, 800);
}
if (game_time < 20) {
    lv_timer_set_period(timer, 600);
}
由于本实验中初始游戏时间是 30 秒，所以游戏开始后地鼠刷新周期会变为 800 ms；当剩余时间小于 20 秒时，又会进一步变为 600 ms。这样设计可以让游戏越到后期刷新越快，使玩家在后半段更紧张，也让游戏难度逐渐增加，而不是从头到尾保持同样节奏
3. 修改地鼠一次出现的数量
在 pop_random_mole() 函数中，程序会先隐藏所有地鼠，然后随机显示一部分地鼠：
for (int i = 0; i < 9; i++) {
    lv_obj_add_flag(moles[i], LV_OBJ_FLAG_HIDDEN);
}
int show_count = rand() % 2 + 1
这里 show_count 的取值范围是 1 到 2，也就是说每次刷新时会随机出现 1 只或 2 只地鼠。如果希望游戏更简单，可以增加每次出现的地鼠数量，例如改成：
int show_count = rand() % 3 + 1
这样一次可能出现 1 到 3 只地鼠，玩家更容易点中目标。但如果刷新速度也同时加快，虽然地鼠数量变多，玩家也需要更快判断位置，游戏仍然会有一定挑战性
如果希望游戏更难，则可以让每次只出现 1 只地鼠，或者降低地鼠出现概率，这样玩家需要更集中注意力，得分也会更困难
4. 修改地鼠显示持续时间
地鼠显示持续时间和 mole_timer 的刷新周期有关。因为每次执行 pop_random_mole() 时，程序都会先把所有地鼠隐藏，再随机显示新的地鼠，所以地鼠大约会显示到下一次刷新为止
也就是说，如果 mole_timer 周期是 1000 ms，地鼠大约显示 1 秒；如果周期改为 600 ms，地鼠显示时间就会明显变短，玩家必须更快点击。这个参数对游戏难度影响很明显，因为玩家的反应时间直接由它决定
如果想降低难度，可以让地鼠显示时间更长；如果想提高难度，可以缩短显示时间，让地鼠更快消失
5. 修改地鼠位置分布
地鼠洞的位置由 hole_positions 数组控制：
static hole_pos_t hole_positions[9] = {
    { 92, 82 }, { 225, 82 }, { 363, 82 },
    { 69, 149 }, { 225, 149 }, { 363, 149 },
    { 69, 219 }, { 225, 219 }, { 382, 219 }
}
这些坐标决定了 9 个地鼠洞在屏幕上的位置。如果地鼠洞之间距离较近，玩家移动手指的距离较短，操作会更轻松；如果地鼠洞分布更分散，玩家需要在屏幕不同区域快速切换，难度会提高
因此，可以通过调整 hole_positions 中的坐标来改变游戏难度。例如把地鼠洞放得更分散，或者放在更靠近屏幕边缘的位置，都会增加玩家点击的难度
6. 修改地鼠大小和点击区域
代码中每个地鼠洞的大小设置如下：
lv_obj_set_size(hole, 80, 60)
地鼠图片是创建在 hole 对象中的：
moles[i] = lv_image_create(hole)
lv_image_set_src(moles[i], R.images.mole)
如果把 hole 的大小调大，或者把地鼠图片显示得更大，玩家更容易点中地鼠，游戏难度会降低。如果把地鼠洞或地鼠图片调小，命中区域变小，玩家需要点得更准确，难度就会提高
另外，代码中还设置了地鼠图片的缩放：
lv_obj_set_style_transform_scale(moles[i], 200, 0)
这个值也会影响地鼠显示大小。通过修改地鼠大小和点击区域，可以比较直观地改变游戏操作难度
7. 修改计分规则
当前代码中，玩家点击到显示状态的地鼠时，分数加 1：
if (!lv_obj_has_flag(mole, LV_OBJ_FLAG_HIDDEN)) {
    score++;
    lv_label_set_text_fmt(score_label, "score: %d", score);
    lv_obj_add_flag(mole, LV_OBJ_FLAG_HIDDEN);
}
这种计分方式比较简单，每打中一只地鼠加 1 分，适合基础实验。如果想让游戏更有挑战性，可以增加新的计分规则。例如：
（1）连续命中多次可以额外加分；
（2）点击空白区域或误点隐藏地鼠时扣分；
（3）剩余时间越少，击中地鼠获得的分数越高；
（4）不同地鼠设置不同分值
这样可以让游戏不只是简单点击，还需要玩家更加准确地判断和操作，游戏策略性也会更强
8. 修改地鼠出现的随机性
代码中使用 rand() 随机选择地鼠出现的位置：
int mole_idx = rand() % 9
这说明地鼠会在 9 个位置中随机出现。如果想让游戏更难，可以避免地鼠连续出现在相近位置，或者让它更容易出现在玩家不容易点击的位置。如果想让游戏简单一些，可以让地鼠更均匀地出现，避免长时间不出现在某些位置
随机性的设计会影响玩家的预判。如果地鼠出现太规律，游戏会变简单；如果完全随机且刷新很快，游戏会更紧张
9. 综合调整方法
如果想降低游戏难度，可以采用以下方法：
将 GAME_TIME 调大，例如从 30 改为 60；
将 mole_timer 周期调大，例如从 1000 ms 改为 1500 ms；
增加每次出现的地鼠数量；
增大地鼠或洞口的点击区域；
取消误触扣分，只保留命中加分
如果想提高游戏难度，可以采用以下方法：
将 GAME_TIME 调小，例如从 30 改为 20；
将 mole_timer 周期调小，例如改为 600 ms；
减少每次出现的地鼠数量；
缩小地鼠尺寸或点击范围；
让地鼠位置更加分散；
加入误触扣分或连续命中奖励机制
总体来看，本游戏的难度可以从时间、速度、数量、位置、大小和计分方式等多个方面进行调整。其中最直接的修改方法是调整 GAME_TIME、mole_timer 的周期以及 show_count 的取值范围。这几个参数改动较小，但对游戏节奏影响明显。通过这些设置，可以把游戏调整成简单、普通或困难等不同难度，更适合不同玩家进行体验！

(十一) 进阶功能2：击中地鼠时播放hit.wav音效

　　  extern int aplay_main(int argc, char *argv[]);
　　  task_create("hit_sound", 100, 81920, aplay_main, argv);
　　}
　　在mole_click_event中，当判断地鼠可见并加分后，调用play_hit_sound()即可在击中地鼠时播放hit.wav。这样实现方式与学习手册中音乐任务的思想一致，都是通过独立任务完成音频播放，避免影响主界面的响应：
　　static void mole_click_event(lv_event_t *e)
　　{
　　  lv_obj_t *mole = lv_event_get_target(e);
　　  LV_LOG_USER("clicked %d", lv_obj_has_flag(mole, LV_OBJ_FLAG_HIDDEN));
　　  if (!lv_obj_has_flag(mole, LV_OBJ_FLAG_HIDDEN))
　　  {
　　    score++;
　　    lv_label_set_text_fmt(score_label, "score: %d", score);
　　    show_hit_effect(mole);
　　    play_hit_sound();
　　    lv_obj_add_flag(mole, LV_OBJ_FLAG_HIDDEN);
　　  }

(十二) 进阶功能3：击中地鼠时LED1闪烁
　　LED1闪烁我参考了开发板使用手册3.3中的LED任务。GPIO设备通过open打开设备节点，再通过ioctl设置GPIO类型和输出电平。通过在串口中输入ls/dev我注意到了：系统中存在gpio0、gpio1等设备节点。进一步查看SDK中的drv_gpio.c可知，当前GPIO映射已经包含LED1和K1，不需要额外修改GPIO驱动：
　　static struct sunxi_gpio_map maps[] = {
　　  {0, GPIOD(21), GPIO_OUTPUT_PIN},         /* LED1 */
　　  {1, GPIOD(7), GPIO_INPUT_PIN_PULLDOWN},  /* KEY1 */
　　  {2, GPIOD(22), GPIO_OUTPUT_PIN},         /* LED2 */
　　  {3, GPIOD(8), GPIO_INPUT_PIN_PULLDOWN},  /* KEY2 */
　　  {4, GPIOD(9), GPIO_INPUT_PIN_PULLDOWN},  /* KEY3 */
　　  {5, GPIOG(12), GPIO_OUTPUT_PIN},         /* WL_REG_ON */
　　};
　　由映射关系可以确定，/dev/gpio0对应LED1，/dev/gpio1对应KEY1，也就是K1。因此本实验中LED1闪烁应操作/dev/gpio0！
　　在MyWhackMole程序中，我封装flash_led1函数。击中地鼠后点亮LED1，短暂延时后再熄灭。由于不同开发板可能存在高电平点亮或低电平点亮的差异，实际实验中，我以LED1能观察到闪烁为准。如果发现亮灭逻辑相反，只需要交换GPIOC_WRITE中的0和1：
　　static void flash_led1(void)
　　{
　　  int fd_led = open("/dev/gpio0", O_RDWR);
　　  if (fd_led < 0)
　　  {
　　    return;
　　  }
　　  ioctl(fd_led, GPIOC_SETPINTYPE, GPIO_OUTPUT_PIN);
　　  ioctl(fd_led, GPIOC_WRITE, 0);
　　  usleep(150 * 1000);
　　  ioctl(fd_led, GPIOC_WRITE, 1);
　　  close(fd_led);
　　}
　　在mole_click_event中，当判断地鼠可见并加分后调用flash_led1()，即可实现击中地鼠时LED1闪烁：
　　static void mole_click_event(lv_event_t *e)
　　{
　　  lv_obj_t *mole = lv_event_get_target(e);
　　  LV_LOG_USER("clicked %d", lv_obj_has_flag(mole, LV_OBJ_FLAG_HIDDEN));
　　  if (!lv_obj_has_flag(mole, LV_OBJ_FLAG_HIDDEN))
　　  {
　　    score++;
　　    lv_label_set_text_fmt(score_label, "score: %d", score);
　　    show_hit_effect(mole);
　　    play_hit_sound();
　　    flash_led1();
　　    lv_obj_add_flag(mole, LV_OBJ_FLAG_HIDDEN);
　　  }
　　}


(十三) 进阶功能4：按下K1键开始游戏
　　按下K1键开始游戏，我同样参考了LED任务中的按键读取逻辑。根据drv_gpio.c中的映射关系，/dev/gpio1对应KEY1，也就是K1。程序可以打开/dev/gpio1作为K1输入设备，通过GPIOC_READ读取按键电平。当检测到K1被按下时，调用start_game函数开始游戏
　　由于start_game会修改LVGL界面对象，而K1检测在线程中运行，因此在线程中调用start_game时应加上LVGL锁，避免和界面刷新同时操作LVGL对象：
　　static void *key_thread(void *arg)
　　{
　　  int fd_key = open("/dev/gpio1", O_RDONLY);
　　  bool key_value;
　　  if (fd_key < 0)
　　  {
　　    return NULL;
　　  }
　　  ioctl(fd_key, GPIOC_SETPINTYPE, GPIO_INPUT_PIN_PULLDOWN);
　　  while (1)
　　  {
　　    ioctl(fd_key, GPIOC_READ, (unsigned long)((uintptr_t)&key_value));
　　    if (key_value == true)
　　    {
　　      lvgl_lock();
　　      start_game(NULL);
　　      lvgl_unlock();
　　      usleep(300 * 1000);
　　    }
　　    usleep(20 * 1000);
　　  }
　　  return NULL;
　　}
　　我在app_create或init_whack_a_mole_game完成界面初始化后，创建key_thread线程。这样游戏启动后可以先停留在准备界面，用户按下K1后再开始游戏；屏幕上的START按钮也可以保留，作为触摸方式开始游戏的备用入口：
　　pthread_t tid_key;
　　pthread_create(&tid_key, NULL, key_thread, NULL);
　　实际测试时发现按下K1没有反应，需要根据串口打印或GPIO读取结果确认按键电平。如果按下时读到0，则将判断条件改为key_value==false；如果按下时读到1，则保持key_value==true。本次SDK中KEY1配置为GPIO_INPUT_PIN_PULLDOWN，因此我默认按下时按高电平触发处理！




五、实验总结
本次实验从驱动修改、应用创建、资源打包、编译烧录到开发板运行验证，完整实现了基于LVGL的WhackMole游戏。相比初探实验，本实验不只是验证任务和线程，而是把LCD显示、触摸屏、LVGL界面、音频、GPIO和按键等内容结合在一起，更接近一个完整的嵌入式交互应用
在基础任务部分，我完成了LCD显示和触摸屏相关配置，使屏幕能够横屏显示，并且触摸坐标能够正常对应游戏界面。随后在示例工程基础上创建MyWhackMole应用，完成Kconfig、Make.defs、Makefile和CMakeLists.txt等文件配置，并编写MyWhackMole_main.c、MyWhackMole.h和MyWhackMole.c，最终成功在开发板上运行打地鼠游戏
在阅读代码和调试过程中，我理解了游戏难度主要由game_time、mole_timer刷新周期、地鼠一次出现数量、地鼠大小和位置、得分规则等因素决定。通过调整这些参数，可以改变游戏节奏和玩家反应压力，也让我对LVGL定时器和事件回调的使用更加熟悉
在进阶任务部分，我给击中地鼠添加了半透明方框特效，让命中反馈更加明显；同时参考openvela手册中的音乐任务和LED任务，实现了击中时播放hit.wav音效、LED1闪烁以及按下K1开始游戏。这样游戏不只是在屏幕上显示，还加入了声音、灯光和物理按键反馈，交互效果更加完整
通过这次实验，我认识到嵌入式图形程序并不是只写界面代码，还要同时考虑驱动配置、资源路径、构建系统、文件系统、设备节点和多任务配合。后续如果继续完善，可以进一步优化音效和LED触发方式，减少阻塞和轮询，也可以加入难度选择、排行榜或更多动画效果!
